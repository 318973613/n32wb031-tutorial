# N32WB031-STB HID 蓝牙拍照遥控器教程

## 1. 项目目标

这个项目的目标很简单：

- 让开发板伪装成一个蓝牙 `HID` 输入设备
- 手机通过系统蓝牙完成配对
- 按下开发板上的按键后，向手机发送“音量键”类控制
- 利用手机相机“音量键拍照”功能，实现蓝牙遥控拍照

## 2. 这个工程实现了什么效果

这份工程上电以后，整体行为如下：

1. 开发板开始广播蓝牙设备
2. OLED 显示当前状态
3. 手机通过系统蓝牙与开发板配对
4. 配对成功后，OLED 显示 `CONNECTED`
5. 按 `KEY1`，开发板向手机发送一个拍照相关的 HID 按键
6. 按 `KEY2`，开发板发送另一个辅助按键

也就是说，这不是一个普通 BLE 透传工程，而是一个**蓝牙输入设备工程**。

## 3. 什么是 HID

`HID` 的全称是 `Human Interface Device`，也就是“人机输入设备”。

最常见的 HID 设备包括：

- 键盘
- 鼠标
- 游戏手柄
- 媒体控制按键

这类设备的特点是：

- 手机或电脑把它们当作“输入源”
- 不需要你自己写协议解析
- 系统会直接接管它们的按键、鼠标、媒体控制行为

所以这份工程的核心思想不是“蓝牙透传一段文本”，而是：

**让手机以为这块开发板就是一个蓝牙遥控器。**

## 4. 为什么拍照比“刷抖音翻页”更容易实现

前面你已经实测过，想直接靠 HID 去“刷抖音翻页”很不稳定，原因是：

- 抖音翻页本质上依赖“触摸滑屏”
- HID 鼠标/滚轮/键盘不一定会被手机当成触摸事件

而拍照就不一样：

- 很多手机相机本来就支持“音量键拍照”
- HID 的媒体控制或音量键输入，更容易被系统接受

所以从工程成功率上看：

- `HID + 拍照`：更现实
- `HID + 抖音翻页`：不稳定

## 5. 工程里用了什么方式来触发拍照

这份工程走的是 **HID Consumer Control** 方向，也就是模拟“音量键”这类媒体控制输入。

当前按键行为设计如下：

- `KEY1(PB1)`：发送 `VOL+`
- `KEY2(PB2)`：发送 `VOL-`

如果你的手机相机设置成“音量键拍照”，那么其中一个键就可以拿来当快门。

你前面已经确认过：

- 你的手机是 **音量下键拍照**

所以测试时优先尝试：

- `KEY2` 作为快门键

## 7. 为什么不能直接用 nRF Connect 测“拍照”

这是这次测试里非常值得写进文章的一个知识点。

很多人刚开始做 BLE，第一反应会用：

- `nRF Connect`
- `LightBlue`

这些 App 确实很适合调试：

- 广播
- GATT 服务
- 特征值
- 通知

但是这份工程是 **HID 输入设备**，不是普通 GATT 透传设备。

所以：

- `nRF Connect` 适合拿来看“设备有没有广播、类型对不对”
- 真正验证拍照，一定要走**手机系统蓝牙配对**

你前面那张图里能看到：

- 设备名 `N32WB031_CAMERA`
- 服务里有 `0x1812`
- `Appearance` 是 `Mouse (HID subtype)`

这说明广播本身已经没问题。  
但真正拍照，不是在 `nRF Connect` 里点，而是在系统蓝牙配对后，让系统把它当作输入设备接管。

## 8. 手机端正确测试流程

这一步非常关键，建议在文章里写得尽量清楚。

正确流程是：

1. 给开发板上电
2. 打开手机系统蓝牙设置
3. 搜索到 `N32WB031_CAMERA`
4. 点击配对
5. 输入密码 `123456`
6. 配对成功后，确认设备已连接
7. 打开手机相机
8. 在相机设置里确认：

   - `音量键功能 = 快门`
   - 或 `音量键拍照`

9. 按开发板按键测试

如果只在 `nRF Connect` 里看到了设备，但没有在系统蓝牙里真正配对成功，那么大概率拍照是不会生效的。

## 9. OLED 在这个工程里的作用

这份工程保留了 OLED，主要是为了让状态更可视化。

它的作用不是“好看”，而是帮助你判断工程到底跑到了哪一步。

OLED 现在会显示这些状态：

- `ADVERTISING`
- `CONNECTED`
- `SHOT / VOL+`
- `AUX / VOL-`

这样即使手机暂时没有拍照成功，你也能先判断：

- 板子有没有广播
- 手机有没有连上
- 按键按下后板子有没有发动作

## 10. 硬件资源与引脚分配

这份工程的主要资源分配如下：

| 功能 | 引脚 | 说明 |
| --- | --- | --- |
| LED1 | `PB0` | 运行状态指示 |
| LED2 | `PA6` | 连接状态指示 |
| KEY1 | `PB1` | 发送 `VOL+` |
| KEY2 | `PB2` | 发送 `VOL-` |
| OLED SCL | `PB8` | 软件 I2C 时钟 |
| OLED SDA | `PB9` | 软件 I2C 数据 |

这里特意保留了：

- `PB6 / PB7`

因为这组引脚在这块板子上和板载 USB 虚拟串口关联最方便，后续调试更顺手。

## 11. 为什么 OLED 改成了软件 I2C

这一步也是一个很好的“踩坑总结”点。

原本如果用硬件 I2C，常见例程会占用：

- `PB6`
- `PB7`

但这组脚位你又想保留给板载 USB 串口那条调试链路，所以这里做了一个更稳的选择：

- `OLED` 改成软件 I2C
- `SCL = PB8`
- `SDA = PB9`

这样做的好处是：

- 串口不冲突
- 接线更灵活
- 文章里也更容易解释为什么这么分配

## 12. 工程内部是怎么工作的

从代码结构上看，这份工程主要分成三部分：

1. `BLE HID`
   - 负责广播、配对、连接和发送 HID 报告

2. `按键检测`
   - 检测 `PB1 / PB2`
   - 做简单去抖

3. `OLED 状态显示`
   - 显示连接状态
   - 显示最近一次动作

换句话说，这个工程不是“手机 App 发命令到板子”，而是“板子主动向手机发 HID 按键”。

## 13. 你这次遇到的问题，为什么会出现

你前面已经遇到过一个很典型的问题：

- 按键后，手机不是拍照，而是在屏幕右下角点了一下、出现聚焦

这个现象的本质是：

- 手机确实收到了 HID 输入
- 但它更像把这个设备当成“鼠标类输入”处理
- 所以落成了“点击聚焦”，而不是“系统音量键快门”

这个问题非常值得写进文章，因为它能体现出：

- 你不是只会“跑通”
- 你是真的在分析手机系统是怎么理解 HID 输入的

后来再改成“更纯粹的拍照遥控器方向”，就是非常合理的迭代过程。

## 15. 常见问题

### 15.1 手机能搜到设备，但不能拍照

优先检查：

- 有没有在系统蓝牙里真正配对，而不是只在 `nRF Connect` 里看见
- 是否输入了密码 `123456`
- 相机里“音量键拍照”是否打开

### 15.2 OLED 显示 CONNECTED，但相机还是没反应

这种情况说明：

- 板子和手机已经连上了
- HID 事件也大概率发出去了

但手机相机 App 不一定接受当前这个 HID 按键映射。  
这时候可以继续尝试：

- `KEY1`
- `KEY2`

看哪个键会被相机识别成快门。

### 15.3 手机出现聚焦，不拍照

这通常说明手机更偏向把它当作鼠标事件处理。  
这也是你前面已经遇到过的真实现象。

### 15.4 iPhone 和安卓会不会不一样

会。

不同品牌、不同系统版本、不同相机 App，对 HID 的兼容性并不完全一样。  
所以这类项目特别适合继续扩展出“兼容性测试”这一篇测评。
