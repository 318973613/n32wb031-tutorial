# N32WB031-STB BLE透传 + OLED 教程

## 1. 工程目标

这份工程的目标很明确：

- 开发板广播一个蓝牙设备名
- 手机连接后，可以像普通 BLE 透传那样收发数据
- 同时把关键状态和最近收到的数据显示在 `0.96 寸 OLED` 上

也就是说，这不是一个“纯蓝牙看手机”的 Demo，而是一个**手机 + 开发板 + OLED 同时联动**的工程。

对应工程目录：

`template/N32WB031_STB_BLE_RDTSS_OLED`

对应 `Keil` 工程：

`template/N32WB031_STB_BLE_RDTSS_OLED/MDK-ARM/N32WB031_STB_BLE_RDTSS_OLED.uvprojx`

`[截图位 1：BLE+OLED 工程目录或工程入口位置]`

## 2. 这个工程实现了什么效果

上电以后，工程的行为大致如下：

1. 开发板开始广播蓝牙设备
2. OLED 显示当前状态为 `ADVERTISING`
3. 手机连接成功后，OLED 状态改成 `CONNECTED`
4. 手机发给开发板的内容，会显示到 OLED 的 `BLE:` 行
5. 串口侧发给开发板的内容，会显示到 OLED 的 `UART:` 行
6. 如果手机打开 `Notify`，串口来的内容还会被回传到手机

从体验上看，你会同时看到三处反馈：

- 手机蓝牙调试 App
- 开发板状态灯
- OLED 屏幕内容

这就是这个工程最有意思的地方，它比普通 BLE 透传更“可视化”。

## 3. 硬件准备

你现在这份工程默认针对最常见的 `0.96 寸 I2C SSD1306 OLED`。

常见 OLED 模块一般有 4 个引脚：

- `VCC`
- `GND`
- `SCL`
- `SDA`

如果你的模块是 `SPI` 版本，这份工程不能直接用。

`[截图位 3：0.96寸 OLED 模块实物图]`

## 4. 引脚分配

这份工程的引脚规划如下：

| 功能 | 引脚 | 说明 |
| --- | --- | --- |
| LED1 | `PB0` | 运行状态指示 |
| LED2 | `PA6` | BLE 连接状态指示 |
| BLE 透传串口 TX | `PB6` | `USART1_TX` |
| BLE 透传串口 RX | `PB7` | `USART1_RX` |
| OLED SCL | `PB8` | 软件 I2C 时钟 |
| OLED SDA | `PB9` | 软件 I2C 数据 |

这里特别要注意一点：

原始 BLE 透传例程默认把 `PB6/PB7` 用作 `USART1` 数据桥接串口。  
考虑到这块开发板板载 USB 虚拟串口正好接到这组引脚上，这一版工程保留了 `PB6/PB7` 作为透传串口。

`[截图位 4：引脚分配总览图或板子标注图]`

## 5. OLED 接线方法

如果你手上的 OLED 是常见 `I2C SSD1306` 模块，可以按下面方式连接：

| OLED 引脚 | 开发板引脚 |
| --- | --- |
| `VCC` | `3.3V` |
| `GND` | `GND` |
| `SCL` | `PB8` |
| `SDA` | `PB9` |

建议优先使用 `3.3V` 供电。  
虽然有些模块标称支持 `5V`，但这块板本身是 `3.3V` 逻辑，直接用 `3.3V` 更稳。

## 7. 为什么这里改成软件 I2C

这是这个工程最关键的一个改动点。

原始透传工程里，`PB6/PB7` 本来就是 `USART1` 透传串口。  
而这块开发板板载 USB 虚拟串口也正好接在这组引脚上，所以它非常适合直接接电脑调试。

这里的处理方式是：

- 透传串口不动，继续保留 `USART1 PB6/PB7`
- OLED 不再使用硬件 `I2C1`
- OLED 改成软件 I2C，放到 `PB8/PB9`

这样最大的好处是：

- 板载 USB 串口还能直接用
- 透传逻辑更接近原始官方例程
- OLED 也能独立工作，不和串口打架

## 8. OLED 显示内容设计

为了让屏幕有实际用途，而不是只显示一个标题，我给 OLED 设计了四行信息：

1. 标题行：`BLE OLED DEMO`
2. 状态行：`STATE: ADVERTISING` 或 `STATE: CONNECTED`
3. BLE 行：最近一条手机发来的内容
4. UART 行：最近一条串口发来的内容

例如连上以后，OLED 上大概会看到：

```text
BLE OLED DEMO
STATE: CONNECTED
BLE: HELLO
UART: OK123
```

这样测试时非常直观，不用只盯着手机。

## 10. OLED 驱动做了什么

这份 OLED 驱动走的是最常见的 `SSD1306 128x64` 流程，但通信层已经改成了**软件 I2C**。  
主要包含：

1. 初始化 `PB8/PB9`
2. 用 GPIO 模拟 `I2C Start/Stop`
3. 用 GPIO 模拟字节发送
4. 发送 SSD1306 初始化命令
5. 维护一个 `8 x 128` 的显存缓冲区
6. 把字符串写到对应行
7. 刷新到屏幕

这份驱动没有追求复杂字体和图形功能，重点是：

- 稳定
- 简单
- 足够看状态和文本

## 11. BLE 数据是怎么显示到 OLED 的

当手机写入 BLE 透传特征值时，数据会先进入 `app_rdtss.c` 的写回调。

这里我已经加了：

```c
oled_set_ble_text(ind_value->value, ind_value->length);
```

这意味着：

- 手机发来的最新一条数据
- 会立刻显示到 OLED 的 `BLE:` 行

所以你在手机里写入文本时，不仅串口侧能收到，OLED 也会同步显示。

![alt text](213f5b4e03ecf13dc673e25e3ff45289.jpg)

## 12. 串口数据是怎么显示到 OLED 的

串口收到数据后，会先进 `app_usart.c` 的接收 FIFO。

这里我加了：

```c
oled_set_uart_text(p_data, len);
```

所以只要串口侧有新数据：

- OLED 的 `UART:` 行就会更新
- 如果手机打开了 `Notify`
- 这段数据还会继续通过 BLE 推回手机

![alt text](image-2.png)

## 13. 连接状态是怎么显示的

在 `app_ble.c` 里，我把 OLED 状态刷新接在了连接和断开回调里：

- 连接成功：`oled_set_conn_state(1);`
- 断开连接：`oled_set_conn_state(0);`

这样：

- 没连上时，OLED 显示 `ADVERTISING`
- 连上以后，OLED 显示 `CONNECTED`

这对调试非常有用，特别是当手机“看起来连了，但你不确定板子状态到底对不对”的时候。

## 14. 手机端怎么测试

手机端推荐用：

- `nRF Connect`
- `LightBlue`

建议测试步骤如下：

1. 给开发板上电
2. 确认 OLED 已亮，并显示 `BLE OLED DEMO`
3. 打开手机 App
4. 搜索设备 `N32WB031_OLED`
5. 点击连接
6. 打开 `Notify`
7. 在可写特征值里发送一段文本，例如 `HELLO`
8. 观察 OLED 是否在 `BLE:` 行显示 `HELLO`
9. 再从串口助手向 `USART1` 发一段文本，例如 `OK123`
10. 观察 OLED 是否在 `UART:` 行显示 `OK123`
11. 同时观察手机是否收到 `Notify`

## 15. 串口端怎么测试

因为透传串口保留在 `USART1`，所以你如果要完整验证双向透传，最方便的方式就是直接走板载 USB 虚拟串口。  
如果你一定要用外部串口模块，那么应该接：

- `PB6`：开发板发出，接串口模块接收
- `PB7`：开发板接收，接串口模块发送

串口参数：

- 波特率：`115200`
- 数据位：`8`
- 停止位：`1`
- 校验位：`None`

## 17. 常见问题

### 17.1 OLED 不亮

优先检查：

- 是不是 `I2C` 版本 OLED，而不是 `SPI` 版
- `VCC / GND / SDA / SCL` 是否接对
- 模块地址是不是 `0x3C`（8 位写法对应 `0x78`）

`[截图位 18：常见错误接线或排查对照图，可选]`

### 17.2 手机能连上，但 OLED 不更新

优先检查：

- 手机有没有真的发到正确的写特征值
- 发送的数据是不是纯文本
- OLED 接线是否正常

### 17.3 手机看不到串口回传

优先检查：

- 有没有打开 `Notify`
- 串口是否真的接在 `USART1 PB6/PB7`
- 串口参数是否 `115200 8N1`

## 18. 这份工程适合拿来做什么

这份 `BLE + OLED` 工程特别适合往下扩展成更有展示感的项目，比如：
